# Human Test Checklist: CI-Fix Arc (v0.7.4 → v0.7.7)

**Target version:** 0.7.7
**Ships with commits:** `ad9489f` (0.7.4), `7ef0078` (0.7.5), and the 0.7.7 commit (this arc's final, which amends the get_path_info fix together with the unctools/cp1252 tooling changes)
**Companion automated tests:** 189 pytest tests (full suite under `tests/`), incl. `tests/test_unc_handler.py` (UNC handling + the `get_path_info` OSError regression), `tests/test_permission_error_handling.py` (async), and the DazzleTreeLib integration tests.

## What this is

A capstone checklist for the modified_datetime_fix CI-fix arc: the four patch releases that took CI from fully red to fully green across the `ubuntu/windows/macos × 3.9–3.12` matrix. The automated suite passes in any environment that has the dependencies; this document covers the surfaces the suite **cannot** see — clean-environment dependency *resolution* (do deps actually install from PyPI?), real `cmd.exe` rendering of the setup scripts, UNC behaviour against an unreachable share, and cross-version/cross-platform install. Run it before tagging a release or whenever the dependency or dev-setup story changes.

## Prerequisites

1. **OS/tooling:** Windows (primary). For full matrix coverage: [`uv`](https://docs.astral.sh/uv/) (provisions Pythons 3.9–3.12) and WSL with a Python 3.9+ (Linux leg).
2. **Isolation:** Every install test runs in a **throwaway venv** — never `pip install` into your daily environment. The pre-flight script creates and tears down its own venv in `%TEMP%`/`/tmp`.
3. **Repo:** a clean checkout/worktree of `folder-datetime-fix` at the version under test. Run all commands from the repo root.
4. **Reset between sections:** each install test makes a fresh venv; delete it after (the pre-flight does this automatically). No persistent state is written outside `%TEMP%`/`/tmp`.

## High-Value Verification (~5 minutes)

If you only have five minutes, run these. If any fail, the release is broken.

### HV.1 — Clean-venv pre-flight is GREEN (the whole arc in one command)

Validates: deps resolve from PyPI + the full suite passes in a CI-faithful environment.

- [ ] **Steps** (same across shells — runs under whatever Python invokes it):
  ```
  python tests/one-offs/preflight_clean_venv.py
  ```
- [ ] **Expected:** ends with `[preflight] GREEN -- safe to push`; `189 passed, 1 skipped`. The `pip show` block lists `dazzle-filekit`, `dazzletreelib`, `dazzle-lib`, `pytest-asyncio` as installed in the throwaway venv.
- [ ] **Why this matters:** this is the gate that would have caught every failure in this arc. If it's red, do not push.

### HV.2 — Runtime deps are actually declared (the 0.7.4 fix)

Validates: `pyproject.toml [project].dependencies` is authoritative; `setup.py` `install_requires` is NOT silently ignored into emptiness.

- [ ] **Steps:** in a throwaway venv, `pip install -e .` then check what came in:

  **cmd.exe**:
  ```cmd
  pip show folder-datetime-fix | findstr /R "Requires"
  ```
  **PowerShell**:
  ```powershell
  pip show folder-datetime-fix | Select-String "Requires"
  ```
  **POSIX**:
  ```bash
  pip show folder-datetime-fix | grep Requires
  ```
- [ ] **Expected:** `Requires: dazzletreelib, dazzle-filekit`. (Pre-0.7.4 this was empty → `ModuleNotFoundError: dazzletreelib` at test collection.)
- [ ] **Why this matters:** the original break — deps declared only where pip doesn't read them.

### HV.3 — `[unc]` extra pulls unctools >= 0.2.0 (the 0.7.7 fix)

Validates: the `[unc]` extra resolves the probe-not-mutate unctools, not a 0.1.x that lacks `classify_path_origin`.

- [ ] **Steps:** in a throwaway venv:
  ```
  pip install -e ".[unc]"
  python -c "import unctools; print(unctools.__version__); print(hasattr(unctools,'classify_path_origin'))"
  ```
- [ ] **Expected:** version `0.2.x` (≥0.2.0), `classify_path_origin` → `True`. UNC enhancement active.
- [ ] **Why this matters:** with the old `>=0.1.0`, pip could install a unctools without `classify_path_origin`; the whole `from unctools import (...)` then fails and UNC support silently disables.

### HV.4 — `setup_dev_environment.bat` renders cleanly in real cmd.exe (the 0.7.7 cleanup)

Validates: no `previous-unc-tests` machine-specific paths; optional-unctools messaging; no codepage garbling.

- [ ] **Steps** (cmd.exe — this is a `.bat`, run it natively):
  ```cmd
  scripts\setup\setup_dev_environment.bat
  ```
  (or read it: `type scripts\setup\setup_dev_environment.bat`)
- [ ] **Expected:** "Installing UNCtools from PyPI…"; on failure it says UNC is *optional* and points at `https://pypi.org/project/unctools/`. **No** reference to `C:\code\previous-unc-tests\...`. No mojibake (ASCII only). "Python 3.9+" (not 3.7+).
- [ ] **Why this matters:** the fallback used to hardcode one developer's machine path; this is a real-subprocess/codepage surface mocks never touch.

### HV.5 — UNC dead-share doesn't crash path inspection (the 0.7.6 fix)

Validates: `get_path_info` degrades gracefully when `resolve()`/`exists()` raise `OSError` (Python 3.9 Windows + unreachable share).

- [ ] **Steps:**
  ```
  python -m pytest tests/test_unc_handler.py::TestUNCPathEdgeCases -v
  ```
- [ ] **Expected:** `test_various_unc_formats` and `test_get_path_info_survives_network_oserror` both PASS. On a machine that truly cannot reach `\\192.168.1.1\share`, the tool must not raise.
- [ ] **Why this matters:** this failed only on the windows-3.9 CI runner — a version+platform+network combination no single local run reproduces.

### HV.6 — Cross-version / cross-platform install (the multi-version gate)

Validates: the arc holds on the Python that failed (3.9) and on Linux.

- [ ] **Steps** (needs `uv` and/or WSL):
  ```bash
  uv python install 3.9
  uv run --no-project --python 3.9 python tests/one-offs/preflight_clean_venv.py
  MSYS_NO_PATHCONV=1 wsl.exe python3 /mnt/c/code/modified_datetime_fix/local/tests/one-offs/preflight_clean_venv.py
  ```
- [ ] **Expected:** both end `[preflight] GREEN`. (Bare `wsl python3 /mnt/...` from Git Bash gets MSYS-path-mangled — the `MSYS_NO_PATHCONV=1` prefix is required.)
- [ ] **Why this matters:** the 0.7.6 bug was Python-3.9-Windows-only; the 0.7.5 UNC bug was Unix-only. A single interpreter on one OS misses both.

## Section 5: The pre-flight tool itself (0.7.5)

## What the automated tests DON'T cover (the point of this doc)

- **Dependency *resolution*** — tests run where deps already exist; they never prove `pip install` pulls them from PyPI (HV.2/HV.3).
- **cmd.exe rendering** of `setup_dev_environment.bat` — codepage, the `if errorlevel` flow, machine-path references (HV.4).
- **Real unreachable-network behaviour** — CI hit `[WinError 64]`; local nets differ (HV.5, 3.2).
- **Cross-version / cross-platform** — one interpreter on one OS missed both the 3.9-only and Unix-only bugs (HV.6).
- **`validate_dev_environment.py` rendered output** — the suggest-fixes text a human reads (4.1).

## Reset / cleanup

- The pre-flight script removes its own venv (`%TEMP%`/`/tmp`) on exit. If interrupted, delete `fdf-preflight-*` dirs there.
- Any manual throwaway venvs: delete the venv directory.
- `uv`-installed Pythons persist (intentional — reusable). No repo state is modified by any step.
